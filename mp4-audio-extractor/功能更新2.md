## 1. 更新概述

### 1.1 更新目标

新增**音频信号分析与可视化**功能。支持对 WAV / FLAC / MP3 音频文件进行频谱、音高、振幅、响度等多维度信号分析，并可选基于用户提供的文字内容进行逐字时序对齐匹配。分析结果以 Canvas 图表形式可视化呈现。

### 1.2 功能范围

| 维度 | 说明 |
|------|------|
| **输入格式** | WAV、FLAC、MP3 |
| **分析维度** | 波形、频谱（热力图）、音高曲线、响度曲线、文字时序对齐 |
| **预处理** | ffmpeg 统一解码为 f32le 16kHz 单声道 PCM |
| **频谱分析** | STFT（短时傅里叶变换），Hann 窗，1024 点, hop 512 |
| **音高检测** | 自相关法基频估计，搜索范围 50-500Hz |
| **响度分析** | 分帧 RMS → dB 转换 |
| **文字对齐** | 能量法语音活动检测 (VAD) → 等比例分配文字至有声段 |
| **可视化** | 5 种 Canvas 图表，时间轴对齐 |
| **交互方式** | 拖拽放入 / 文件对话框两种方式 |

### 1.3 设计原则

- **页面隔离**：分析功能作为独立子视图，通过主页面导航卡片进入，返回按钮退出
- **零外部图表库**：全部 Canvas 手写渲染，不引入 Chart.js / ECharts 等依赖
- **算法自包含**：DFT / 自相关 / Hann 窗 / VAD 全部在 Rust 中手写实现
- **信号标准化**：所有音频统一重采样至 16kHz 单声道 PCM f32，确保算法参数恒定
- **可选文字输入**：不填文字时执行纯信号分析，填写后额外触发语音段切割与文字对齐

## 2. 需求分析

### 2.1 用户故事

> 作为用户，我手头有一段语音录音（可能来自视频提取），我想直观地看到它的波形、频谱成分、音高变化和响度分布。如果我有这段录音的文本稿，我希望能看到每个文字大致出现在哪个时间点上。

### 2.2 功能边界

- ✅ 支持 WAV / FLAC / MP3 三种格式作为输入
- ✅ 自动分析并渲染：波形图、频谱热力图、音高曲线、响度曲线
- ✅ 可选填文字：程序自动切割语音段并按字数等比例分配时序
- ✅ 拖拽放入 + 文件对话框双输入方式
- ✅ Canvas 图表全部附带时间轴刻度
- ❌ 不支持实时录音分析（仅离线文件）
- ❌ 不支持多声道独立分析（统一降混为单声道）
- ❌ 不保存图表为图片文件（仅屏幕渲染）
- ❌ 文字对齐为估算（等比例分配，非基于 ASR 引擎）

### 2.3 与其他功能的差异

| 维度 | 视频提取 | 音频转换 | 音频分析 |
|------|----------|----------|----------|
| 输入类型 | `.mp4` | `.wav/.flac/.mp3` | `.wav/.flac/.mp3` |
| 核心操作 | ffmpeg 提取音频轨 | ffmpeg 格式转换 | DSP 信号处理 + Canvas 渲染 |
| 输出形式 | 文件 (.wav) | 文件 (.wav/.flac/.mp3) | 可视化图表（5 种） |
| UI 模式 | 主页面内嵌卡片 | 主页面内嵌卡片 | 独立子视图（导航跳转） |
| 是否需要额外输入 | 否 | 是（目标格式） | 可选（文字稿） |

## 3. 技术选型

### 3.1 信号处理算法

| 分析项 | 算法 | 参数 | 理由 |
|--------|------|------|------|
| 重采样 + 解码 | ffmpeg f32le | 16000Hz, 1ch | 统一预处理，降低后续算法复杂度 |
| 波形降采样 | 分块 RMS+Peak 混合 | 目标 1200 点 | 平衡绘制精度与数据量 |
| 短时傅里叶变换 | STFT + 手写 DFT | Hann 窗 1024 点, hop 512 | Hann 窗减少频谱泄漏 |
| 频谱图 | DFT 幅度 → 128 频段 | 对数量化 | 模拟人耳感知，降低渲染像素数 |
| 音高 | 自相关法 (Autocorrelation) | Lag 范围 32-320 (50-500Hz) | 时域方法，无需 FFT，适合单音高检测 |
| 响度 | RMS → dB | 分帧 1024 点, hop 512 | `20·log₁₀(RMS)` |
| 语音活动检测 (VAD) | 能量阈值法 | 5% 最大能量，最小段 50ms | 简单有效，适合干净的语音录音 |
| 文字对齐 | VAD 段等比例分配 | 按字符数平分有声时长 | 无需 ASR，轻量实现 |

### 3.2 为什么选择手写 DFT 而非 FFT

| 决策点 | DFT | FFT（如 RustFFT） |
|--------|-----|-------------------|
| 额外依赖 | 零 | 需要引入 `rustfft` 等 crate |
| 性能 | 1024 点 DFT 为 O(n²)=1M 次运算/帧，音频数秒内可完成 | O(n log n)，更快 |
| 代码量 | ~10 行 | 需要适配 FFT planner |
| 选型结论 | 对于本工具的用户场景（数秒至数十秒音频），DFT 性能完全可接受 | 弃用 |

> 实测：30 秒 16kHz 音频 ≈ 480,000 samples，约 938 帧（1024 点 / hop 512），每帧 DFT 约 1M 次乘加，总计约 9.4 亿次运算。在现代 CPU 上可在 1-2 秒内完成。

### 3.3 音高检测选型

| 方法 | 原理 | 优点 | 缺点 | 选型 |
|------|------|------|------|------|
| 自相关 | 信号自身的延迟相关性 | 时域计算，直觉清晰 | 对噪音敏感 | ✅ 选用 |
| 倒谱 (Cepstrum) | FFT → log → IFFT | 分离激励/声道 | 需要 FFT | ❌ |
| YIN 算法 | 差分函数改进自相关 | 业界标准，精度高 | 实现复杂 | ❌ |

### 3.4 VAD 选型

| 方法 | 原理 | 选型 |
|------|------|------|
| 能量阈值 | RMS 超过 max×5% 视为语音 | ✅ 选用（简单，对干净录音有效） |
| ZCR（过零率） | 清音/浊音过零率差异大 | ❌ 需要结合能量使用 |
| WebRTC VAD | 高斯混合模型 | ❌ 需要额外库 |

### 3.5 Canvas 图表配色方案

每种子图表使用独立配色，在暗色背景上清晰可辨：

| 图表 | 主色 | 含义 |
|------|------|------|
| 📊 波形图 | `#6c63ff`（主题紫） | 振幅随时间变化 |
| 🌈 频谱图 | 蓝→紫→红渐变（`putImageData`） | 频率能量分布 |
| 🎹 音高曲线 | `#22d3bb`（青绿） | 基频 H z 随时间变化 |
| 🔊 响度曲线 | `#eab308`（金色）面积 + 轮廓 | dB 随时间变化 |
| 📝 文字时序 | 6 色循环（紫/青/金/红/绿/蓝） | 每字/每段颜色区分 |

### 5.3 M1: extract_pcm() — 音频预处理引擎

```rust
fn extract_pcm(input_path: &str) -> Result<(Vec<f64>, f64, u32), String>
```

**职责**：将任意 WAV/FLAC/MP3 音频统一解码为 f32le 16kHz 单声道 PCM。

**执行流程**：

```
┌─[1] ffprobe 获取元数据 ────────────────────────────────┐
│ ffprobe -v error -select_streams a:0                    │
│   -show_entries format=duration:stream=sample_rate,     │
│   channels -of default=noprint_wrappers=1               │
│ 返回值: duration, sample_rate, channels                 │
└────────────────────────────────────────────────────────┘
                         │
┌─[2] ffmpeg 统一解码 ───────────────────────────────────┐
│ ffmpeg -i {input} -f f32le -acodec pcm_f32le            │
│   -ar 16000 -ac 1 -y pipe:1                             │
│                                                         │
│ 参数说明:                                                │
│   -f f32le     → 输出格式：32-bit float little-endian   │
│   -acodec f32le→ 编码为浮点 PCM                         │
│   -ar 16000    → 重采样至 16kHz                         │
│   -ac 1        → 强制单声道                             │
│   pipe:1       → 输出到 stdout（避免临时文件）           │
└────────────────────────────────────────────────────────┘
                         │
┌─[3] 字节 → f64 数组 ───────────────────────────────────┐
│ stdout 字节流 → chunks_exact(4) → f32::from_le_bytes    │
│   → as f64 → Vec<f64>                                   │
└────────────────────────────────────────────────────────┘
```

**设计决策：为什么是 16kHz 单声道？**

| 参数 | 值 | 理由 |
|------|-----|------|
| 采样率 | 16000Hz | 人声基频范围 85-255Hz，16kHz 可覆盖 0-8kHz（Nyquist），足够语音分析 |
| 位深 | f32 | 避免整数 PCM 的量化噪声，浮点计算精度更高 |
| 声道 | 1（mono） | 分析统一处理单声道，简化算法 |

### 5.4 M2: compute_dft() — 手写离散傅里叶变换

```rust
fn compute_dft(input: &[f64]) -> Vec<f64>
```

**数学定义**：

$$
X[k] = \frac{1}{N} \sqrt{ \left( \sum_{n=0}^{N-1} x[n] \cdot \cos\left(\frac{-2\pi k n}{N}\right) \right)^2 + \left( \sum_{n=0}^{N-1} x[n] \cdot \sin\left(\frac{-2\pi k n}{N}\right) \right)^2 }
$$

**实现要点**：
- 仅计算前 N/2 个频点（对称性，后半为镜像）
- 返回归一化幅度值（除以 N）
- 时间复杂度 O(N²) ，N=1024 时每帧约 100 万次操作

### 5.5 M3: compute_autocorrelation() — 自相关

```rust
fn compute_autocorrelation(data: &[f64], max_lag: usize) -> Vec<f64>
```

**数学定义**：

$$
R[k] = \frac{1}{N-k} \sum_{n=0}^{N-k-1} x[n] \cdot x[n+k]
$$

用于音高检测：自相关函数的峰值位置对应基音周期，`pitch_hz = sample_rate / best_lag`。

### 5.6 M4: hann_window() — Hann 窗函数

```rust
fn hann_window(size: usize) -> Vec<f64>
```

**公式**：

$$
w[n] = 0.5 \cdot \left(1 - \cos\left(\frac{2\pi n}{N-1}\right)\right)
$$

在 STFT 中加窗可减少频谱泄漏（sidelobe 衰减约 -31.5dB/oct，优于矩形窗的 -6dB/oct）。

### 5.7 M5: compute_spectrogram() — 频谱图

```rust
fn compute_spectrogram(samples, sample_rate, window_size, hop)
    -> serde_json::Value
```

**处理流程**：

```
samples[0..N]
  │
  ├─ 滑动窗口: 每 hop=512 点取 window_size=1024 点
  │    ├─ 加 Hann 窗
  │    ├─ compute_dft() → 512 个频点幅度
  │    └─ 降采样至 128 个频段（减少数据量）
  │
  └─ 输出 JSON:
     {
       "times": [0.0, 0.032, 0.064, ...],
       "frequencies": [0, 62.5, 125, ...],  // 128 个对数分布频点
       "magnitudes": [[0.001, 0.03, ...], ...]  // 每帧 128 个值
     }
```

### 5.8 M6: compute_pitch() — 音高检测

```rust
fn compute_pitch(samples, sample_rate, window_size, hop)
    -> serde_json::Value
```

**执行流程**：

```
每帧 (1024 点, hop 512):
  ├─ compute_autocorrelation(frame, max_lag)
  │   max_lag = 16000 / 50 = 320 (对应 50Hz)
  │   min_lag = 16000 / 500 = 32  (对应 500Hz)
  │
  ├─ 在 [min_lag, max_lag) 范围内找自相关最大值
  │   ├─ 峰值 < 0.0001 → 无声（pitch = 0）
  │   └─ 峰值 ≥ 0.0001 → pitch_hz = sample_rate / best_lag
  │
  └─ 输出: { times: [...], values: [...] }
```

**音高 0 的含义**：表示当前帧为无声段（环境静音或辅音），前端绘制时会自然落到 Y=0 处。

### 5.9 M7: compute_loudness() — 响度

```rust
fn compute_loudness(samples, sample_rate, window_size, hop)
    -> serde_json::Value
```

**公式**：

$$
\text{RMS} = \sqrt{\frac{1}{N}\sum_{n=0}^{N-1} x[n]^2}
$$

$$
\text{dB} = 20 \cdot \log_{10}(\text{RMS})
$$

当 RMS = 0 时返回 -100dB（避免 `log(0)` 的 -∞）。

### 5.10 M8: compute_waveform() — 波形降采样

```rust
fn compute_waveform(samples: &[f64], target_points: usize) -> Vec<f64>
```

**算法**：将全部采样点等距分为 `target_points`（1200）个块，每块取 RMS 与 Peak 的平均值。这种混合策略比单纯取峰值或平均值更能反映波形的视觉特征。

### 5.11 M9: energy_based_vad() — 能量法语音段检测

```rust
fn energy_based_vad(samples, sample_rate, window_size, hop)
    -> Vec<(f64, f64)>
```

**执行流程**：

```
1. 分帧计算 RMS 能量
2. 取最大能量的 5% 作为阈值
3. 遍历帧: RMS > 阈值 → 语音开始; RMS ≤ 阈值 → 语音结束
4. 过滤 < 50ms 的碎片段
5. 合并间隔 < 100ms 的相邻段
6. 边界情况: 若不间断语音覆盖 > 90% 时长 → 整段返回
```

### 5.12 M10: align_text_to_segments() — 文字时序对齐

```rust
fn align_text_to_segments(segments, transcript)
    -> Vec<serde_json::Value>
```

**核心算法**：

```
1. 剥离文本中的空白字符，得到字符列表 [c₁, c₂, ..., cₙ]
2. 计算总有声时长 total_speech = Σ(seg[i].end - seg[i].start)
3. 每个字符分配时长 ≈ total_speech / n
4. 逐段填充字符:
   对于每个 VAD 段:
     从当前段起始时间开始，按分配时长依次放置字符
     超出段尾则切换到下一段
5. 输出: [ { text: "你", start: 0.12, end: 0.18 }, ... ]
```

**设计说明**：这是一个**估算方法**，假设说话速度均匀——在没有 ASR 引擎的情况下，这是最合理的近似方案。

### 5.13 M11: analyze_audio() — 分析编排器

```rust
#[tauri::command]
fn analyze_audio(input_path: String, transcript: Option<String>)
    -> Result<String, String>
```

**编排流程**：

```
┌─[1] 输入校验 ──────────────────────────────────────────┐
│ 文件是否存在? 格式是否在 {"wav","flac","mp3"} 中?       │
└────────────────────────────────────────────────────────┘
                         │
┌─[2] extract_pcm() ─────────────────────────────────────┐
│ ffmpeg 解码 → (samples, duration, sample_rate)          │
└────────────────────────────────────────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼              ▼
   waveform()    spectrogram()   pitch()    loudness()
   (1200点)      (1024点 STFT)  (自相关)   (RMS→dB)
                         │
          (if transcript is Some)
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼
      VAD()        align_text_to_segments()
                         │
┌─[3] JSON 序列化 ───────────────────────────────────────┐
│ serde_json::json!({ duration, waveform, spectrogram,     │
│   pitch, loudness, text_alignment })                    │
└────────────────────────────────────────────────────────┘
```

### 6.6 M23-M27: Canvas 图表渲染

#### 6.6.1 通用初始化函数

```javascript
function initCanvas(id) {
  const canvas = document.getElementById(id);
  const rect = canvas.getBoundingClientRect();
  const dpr = window.devicePixelRatio || 1;
  canvas.width = rect.width * dpr;    // 物理像素
  canvas.height = rect.height * dpr;
  const ctx = canvas.getContext("2d");
  ctx.scale(dpr, dpr);               // 缩放至 CSS 像素
  ctx.clearRect(0, 0, rect.width, rect.height);
  return { ctx, w: rect.width, h: rect.height };
}
```

**关键**：使用 `devicePixelRatio` 缩放确保在 HiDPI 屏幕上清晰渲染。

#### 6.6.2 M23: 波形图 (renderWaveform)

| 属性 | 值 |
|------|-----|
| 数据 | `data.waveform` (1200 点，归一化至 [-1, 1]) |
| 绘制方式 | `ctx.moveTo/lineTo` 折线图 |
| 中线 | 虚线 `rgba(108,99,255,0.3)` 贯穿中部 |
| Y 轴范围 | 信号幅度 [-1, 1]，映射至 [h×0.05, h×0.95] |
| 时间轴 | 底部 5 等分刻度 |

#### 6.6.3 M24: 频谱图 (renderSpectrogram)

| 属性 | 值 |
|------|-----|
| 数据 | `data.spectrogram.magnitudes` (times × 128 矩阵) |
| 绘制方式 | `ctx.createImageData` → 逐像素赋色 → `putImageData` |
| 颜色映射 | 蓝（低能量）→ 紫 → 红（高能量） |
| Y 轴 | 频率 0-8kHz（对数分布），底部=低频，顶部=高频 |

**像素赋色公式**：
```javascript
const r = Math.min(val * 2, 1);
const g = Math.min(val * 1.2, 0.6);
const b = Math.min(1 - val, 0.8);

pixel[0] = Math.floor(r * 80 + b * 120);   // R: 高能偏红, 低能偏蓝
pixel[1] = Math.floor(g * 40 + val * 100);  // G: 中等能量过渡
pixel[2] = Math.floor(b * 100 + r * 155);   // B: 高能偏紫, 低能偏蓝
pixel[3] = 255;                              // A: 不透明
```

#### 6.6.4 M25: 音高曲线 (renderPitch)

| 属性 | 值 |
|------|-----|
| 数据 | `data.pitch.values` (Hz，0 表示无声) |
| 绘制方式 | `ctx.moveTo/lineTo` 折线，`#22d3bb` 青绿色 |
| Y 轴范围 | 0-500Hz |
| Y 轴刻度 | 每 100Hz 一格 |

#### 6.6.5 M26: 响度曲线 (renderLoudness)

| 属性 | 值 |
|------|-----|
| 数据 | `data.loudness.values` (dB，约 -100 到 0) |
| 绘制方式 | 面积填充 `ctx.fill()` + 轮廓线 `ctx.stroke()` |
| 填充色 | `rgba(234,179,8,0.25)` 金色半透明 |
| 轮廓色 | `#eab308` 金色实线 |
| Y 轴范围 | 动态（根据实际最小值 → 0dB） |
| Y 轴刻度 | 4 格 dB 标注 |

#### 6.6.6 M27: 文字时序 (renderTextAlignment)

| 属性 | 值 |
|------|-----|
| 数据 | `data.text_alignment[]` 每项含 `{text, start, end}` (秒) |
| 绘制方式 | 彩色 `fillRect` 分段矩形 + 白色文字标注 |
| 配色 | 6 色循环：紫/青/金/红/绿/蓝 |
| 合并规则 | 相邻同文字合并为一个连续段 |
| 最小可视宽度 | 段宽 < 20px 不显示文字标签 |

**仅当用户填写了文字且分析返回了文字时序数据时**，文字时序图表才显示，否则整块隐藏。

## 8. 整体协调性保障

### 8.1 模块间契约

| 模块 | 输入 | 输出 |
|------|------|------|
| `extract_pcm` | 文件路径 | `(Vec<f64>, f64, u32)` |

| `compute_autocorrelation` | `&[f64]` + max_lag | `Vec<f64>`（自相关系数） |
| `hann_window` | usize | `Vec<f64>`（窗系数） |
| `compute_spectrogram` | `&[f64]` + 参数 | JSON: `{times, frequencies, magnitudes}` |
| `compute_pitch` | `&[f64]` + 参数 | JSON: `{times, values}` |
| `compute_loudness` | `&[f64]` + 参数 | JSON: `{times, values}` |
| `compute_waveform` | `&[f64]` + target_points | `Vec<f64>` |
| `energy_based_vad` | `&[f64]` + 参数 | `Vec<(f64, f64)>` |
| `align_text_to_segments` | segments + transcript | `Vec<Value>` |
| `analyze_audio` | `(String, Option<String>)` | `Result<String, String>` (JSON) |

### 8.4 Canvas 渲染协调

所有 Canvas 图表共用同一套时间轴对齐规则：

```
所有图表 X 轴 = 时间 (0 → duration 秒)
├─ 波形图:       time → 1200 个均匀采样点
├─ 频谱图:       time → spectrogram.times (约 duration/0.032 个时间窗)
├─ 音高曲线:     time → pitch.times (同上)
├─ 响度曲线:     time → loudness.times (同上)
└─ 文字时序:     time → text_alignment[].start / .end
```

时间轴刻度统一为 5 等分（频谱/音高/响度）或 6 等分（文字时序），单位统一为秒（s），格式保留 1 位小数。

### 8.5 内存与性能

| 项目 | 估算 |
|------|------|
| 30 秒 16kHz PCM | 480,000 f64 × 8 bytes = ~3.8 MB |
| DFT (1024 点, 938 帧) | ~938 × 1M 次运算 ≈ 9.4 亿次浮点，< 2 秒（现代 CPU） |
| 频谱 JSON | 938 帧 × 128 频点 × ~4 bytes ≈ 480 KB 文本 |
| Canvas 总像素 | 820×180×4×5 + 820×180×4 = ~3.5 MB 显存 |

### 8.6 错误处理统一

所有错误通过 Tauri 的 `Err(String)` 返回，前端通过 `try/catch` 处理：

```
视频提取错误 → showStatus("error", msg)      → 红色提示条
音频转换错误 → showConvertStatus("error", msg) → 红色提示条
音频分析错误 → showAnalysisStatus("error", msg) → 红色提示条
```

三种错误展示使用完全相同的 CSS 样式（`.status.error`），视觉一致。
